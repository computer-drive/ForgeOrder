# API设计
本文档介绍ForgeOrder的网页API设计规范。

## 基本原则
对于所有接口，URL都应以`/api/`开头。

对于所有接口，请求体（若有）、响应体都应使用json格式。

对于需要传递参数的接口，应使用请求体而不是URL参数。

对于所有接口的名称，应以RPC风格的方式，使用小驼峰命名法命名。对于接口名称中出现的动词，应遵守以下原则：
 - 使用`new`而不是`create`或其他名称
 - 使用`update`而不是`edit`或其他名称

## 请求原则
请求体根据接口的参数定义，键名为字符串，键值可为任意类型的值，但应与后端定义的参数类型相匹配。

对于需要认证的接口，请求头中务必包含`Authorization`字段，其中包含Token的值。

## 响应原则
响应体必须包含以下三个字段：
 - `status`：响应状态，整数；
 - `data`：响应数据。

`status`为本次请求的状态，不同的`status`值代表了不同的状态。

响应体未设置`message`字段，尽量不使用字符串等自然语言描述状态。

`status`的值一般为`0`或一个四位正整数，在千位的不同的数字代表不同的错误类型，如下表：

|`status` | 错误类型 | 说明 |
| --- | --- | --- |
| `0` | 请求成功 | 无 |
| `1xxx` | 一般错误 | 请求时发生的一般错误，通常为客户端的问题 |
| `2xxx` | 授权/认证错误 | 用户认证、权限问题 |
| `3xxx` | 逻辑错误 | 各接口的具体错误 |
| `9xxx` | 系统错误 | 服务端错误 |

请注意，`3xxx`的开头的状态码为各路由的具体错误，同一状态码在不同的路由下所代表的含义是不同的。在接下来的章节中，会介绍各路由的通用错误，以及各路由的具体错误。

### 通用错误

不同的`status`一般还拥有特殊的HTTP状态码。

#### 以`1`开头的`status`
| `status` | HTTP状态码 | 说明 |
| --- | --- | --- |
| `1000` | `400` | 未知的客户端错误 |
| `1001` | `400` | 参数验证失败 |
| `1002`| `405` | 请求的方法错误 |
| `1003`| `404`| 请求的接口不存在 |
| `1004` | `429` | 请求的频率过高 |

#### 以`2`开头的`status`
| `status` | HTTP状态码 | 说明 |
| --- | --- | --- |
| `2001` | `401` | 用户未登录 |
| `2002`| `401`| 用户权限不足（非管理员用户访问了一个管理员页面）|
| `2003`| `401` | Token无效 |
| `2004`| `401` | Token过期|
| `2005` | `401` | 旧设备登录 |

#### 以`9`开头的`status`
| `9001` | `500` | 服务器内部错误 |
| `9002` | `500` | 数据库错误 |
| `9003` | `500` | 文件系统错误 |
| `9004` | `500` | 其他系统错误 |


### 通用错误的具体说明

#### 1001
参数验证失败时的错误，参数验证逻辑请参见[Args Verify](./server/args_verify.md)。

本错误中，`data`字段将包含参数验证的详细信息，例如：
```json
{
    "status": 1001,
    "data": {
        "arg1": "NOT_FOUND",
        "arg2": "TYPING_ERROR"
    }
}
```

其中`NOT_FOUND`表示必填项缺失，`TYPING_ERROR`表示参数的类型错误。




#### 2001
访问一个需要认证的接口但请求头中无`Authorization`字段将返回本错误。
